The Netlify deploy errored, with the following guidance provided:

Diagnosis
- The install failed because pnpm is running with a frozen lockfile and the lockfile is out of sync with package.json. See the frozen-lockfile error at [line 21](#L21).
- The log shows the concrete mismatch: a dependency was removed in package.json but still present in pnpm-lock.yaml: [line 24](#L24).
- Netlify exits the build due to the install failure: [line 26](#L26) and the final failure lines [line 26-L28](#L26-L28).

Cause
- package.json (or workspace package.json) and the committed pnpm-lock.yaml are inconsistent. In this case @workspace/api-client-react was removed/changed in package.json but pnpm-lock.yaml still references it, and CI runs with frozen-lockfile enabled by default so installation is blocked.

Solution
Preferred fix — update and commit the lockfile
1. Verify the package change is intentional: check your root and workspace package.json files to confirm @workspace/api-client-react was removed (or should still exist).
2. On your local machine (at the repo root) run:
```bash
pnpm install
# or, if you only want to refresh the lockfile without installing node_modules:
pnpm install --lockfile-only
```
This will update pnpm-lock.yaml to match package.json.
3. Commit and push the updated lockfile:
```bash
git add pnpm-lock.yaml
git commit -m "Update pnpm-lock.yaml to match package.json"
git push
```
4. Re-trigger the Netlify build.

Alternative temporary/workaround — allow CI to ignore frozen-lockfile
- If you cannot update and commit the lockfile right away, you can instruct CI to run without frozen-lockfile. Two options:
  - Change your Netlify build command to run pnpm without frozen lock:
    ```bash
    pnpm install --no-frozen-lockfile
    # then continue your normal build command
    ```
  - Or set the PNPM_FLAGS environment variable in Netlify build settings:
    PNPM_FLAGS="--no-frozen-lockfile"

Notes
- The correct long-term fix is to keep pnpm-lock.yaml committed and in sync with package.json; disabling frozen-lockfile in CI hides the underlying inconsistency.
- If you are using workspaces, run the commands at the repository root so the workspace lockfile is regenerated correctly.
