- Relevant log lines
  - The install step is running with NODE_ENV=production: [line 17](#L17).
  - pnpm fails because the lockfile is out of date and CI is using a frozen lockfile: [line 22](#L22).
  - pnpm reports which dependency/specifier mismatch caused the failure: [line 25](#L25).
  - Your custom build command included --no-frozen-lockfile but Netlify runs dependency install before the build command, so that flag in the build command does not affect Netlify’s install step: [line 9](#L9).

Diagnosis
- Error type: pnpm install failure due to an out-of-sync pnpm-lock.yaml (ERR_PNPM_OUTDATED_LOCKFILE).
- Cause: pnpm in CI enforces a frozen lockfile by default. The repo’s pnpm-lock.yaml does not match the package specifiers (one workspace dependency @workspace/api-client-react was removed/changed), so pnpm refuses to install. The build’s attempt to pass --no-frozen-lockfile in the build command does not help because Netlify runs its own install step before the build command.

Solution (pick one of the two preferred fixes; 1 is recommended)
1) Update and commit the lockfile locally (recommended)
   - Verify the workspace/package is present and correct:
     - Confirm that the package specifier mentioned in the error ( @workspace/api-client-react ) is intentionally present or removed in your workspace package.json files. Check your repository package.json(s): https://github.com/NerochazeDev/Nerochaze-tools/blob/main/package.json
     - If you removed or renamed that workspace package, ensure the package files/manifest were committed and workspace config was updated.
   - On your local machine, from the repo root run:
```bash
corepack enable
pnpm install
```
   - This updates pnpm-lock.yaml to match package.json. Commit and push the updated lockfile:
```bash
git add pnpm-lock.yaml
git commit -m "Update pnpm-lock.yaml to match package.json"
git push
```
   - Netlify will use the updated lockfile and the install should succeed.

2) Temporary/alternate Netlify-side fixes (if you cannot update the lockfile immediately)
   - Option A — Allow non-frozen install on Netlify:
     - In Netlify site settings > Build & deploy > Environment > Environment variables, add:
       - Key: PNPM_FLAGS
       - Value: --no-frozen-lockfile
     - Or set the Install command in Netlify to explicitly run pnpm without frozen lockfile (Project settings > Build & deploy > Build settings > Edit “Install command”):
```bash
corepack enable && pnpm install --no-frozen-lockfile
```
     - Note: This is a workaround. The proper fix is to keep lockfile committed and in sync.
   - Option B — Ensure devDependencies are available during install (if missing devDeps cause the mismatch):
     - The log shows NODE_ENV=production ([line 17](#L17)) so devDependencies will not be installed. If your workspace relies on devDependencies during install, either:
       - Temporarily set NODE_ENV to blank or 'development' in Netlify environment variables during install, or
       - Make dev-only packages true dependencies, or
       - Preferably update the lockfile so install succeeds with NODE_ENV=production.

Why this will fix it
- pnpm enforces lockfile correctness in CI; updating and committing pnpm-lock.yaml synchronizes the specifiers so pnpm can install deterministically. Allowing --no-frozen-lockfile on Netlify only bypasses the check but does not address the underlying mismatch, so updating the lockfile is the recommended long-term fix.
